# РЕШЕНО: двухфазное подтверждение хода (Вариант A + Redis)

## Контекст

Системный промпт геймместера (docs/02) предполагает: если решение рискованное,
ИИ возвращает `advisor_objection`, который должен быть показан игроку
**до** того, как ход применится — игрок может пересмотреть решение.

## Принятое решение

**Вариант A — двухфазный эндпоинт, с Redis для временного хранилища.**

```
POST /games/:id/turns/preview
  -> вызывает ИИ, считает дельты через rules-engine (но не применяет),
     сохраняет классификацию в Redis с TTL 5 минут
  -> возвращает narrative + advisorObjection + statDeltasPreview игроку

POST /games/:id/turns/confirm
  -> читает pending-классификацию из Redis
  -> применяет rules-engine, пишет в Postgres в транзакции
  -> очищает Redis-запись
  -> 409, если pending не найден (TTL истёк) или current_turn рассинхронизирован

POST /games/:id/turns/cancel
  -> игрок передумал — просто чистит Redis-запись
```

## Почему Redis, а не Postgres-поле или in-memory

Redis выбран не как временная затычка, а как инфраструктура, которая всё
равно понадобится на этапе 5 (мультиплеер — presence, синхронизация комнат).
Вводим её раньше по времени, не добавляя новую сущность специально под это.

Реализация: `backend/src/db/pending-turns.js` — простая обёртка
`save/get/clear` с TTL через `SET ... EX`.

## Что проверено тестами (сквозной прогон с моками Postgres/Redis/Claude API)

1. **Preview не открывает Postgres-транзакцию** — только `db.query()` для
   чтения текущего состояния, никакого `db.connect()`/`FOR UPDATE`.
2. **statDeltas совпадают 1:1 между preview и confirm** — тот же seed
   (`gameId:turnNumber:action_type`) даёт идентичный результат rules-engine
   при обоих вызовах. Игрок видит ровно то, что применится.
3. **Защита от рассинхрона**: если `current_turn` в БД успел измениться
   между preview и confirm (например параллельный запрос), confirm вернёт
   409 и попросит сделать preview заново — вместо того чтобы применить
   устаревшие дельты.
4. **Confirm без предварительного preview** → 409 с понятной ошибкой,
   а не падение или тихое создание хода из ничего.

## Найденный и исправленный баг в процессе тестирования

Первая версия `pending-turns.js` деструктурировала только
`{ gmClassification, turnNumber }` в `save()`, отбрасывая остальные поля
payload (`statsAfterDelayed`, `remainingEffects`). Это привело бы к падению
`confirm` на проде с `TypeError: Cannot read properties of undefined`.
Исправлено: `save()` теперь сохраняет весь переданный объект целиком.
Это наглядный пример того, зачем нужен сквозной тест, а не только
юнит-тесты отдельных модулей — баг был на стыке двух файлов.

## Что это меняет в frontend

`App.jsx`: кнопка "Огласить" стала двухшаговой — "Рассмотреть" (preview)
показывает карточку с нарративом, предполагаемыми дельтами и возражением
советника (если есть), внутри которой уже отдельные кнопки "Огласить"
(confirm) и "Пересмотреть решение" (cancel). `api.js` обновлён под три
новых эндпоинта вместо одного `submitTurn`.
